## 문서 구성

1. **팀 정보** — 팀명/팀원/선택 산업분야(AI 개발 플랫폼)
2. **문제 후보 도출 (3개)**
   - ① 학교/강의 자료 관리의 비효율성 — 자료가 여러 플랫폼에 흩어져 탐색·준비에 반복 작업
   - ② **팀 프로젝트 협업의 어려움** — 역할 분담·일정·진행 공유가 수작업 → 지연·혼선
   - ③ AI 활용 경험·실습 기회 부족 — 범용 LLM뿐, 맞춤형 앱 제작·실험 기회 제한
3. **최종 문제 선택 — 2번** — 중요 이유(프로젝트 성과 직결·협업 학습 기회) + 기존 해결방법의 한계(오프라인 회의/공유문서/범용 PM툴 모두 학교 데이터와 미연결)
4. **아이디어 스케치 — 동명대 자체 AI 프로젝트 도우미**
   - 캠퍼스 프로젝트 도우미 AI (진행상황 실시간 관리·제출 마감 알림·역할 배분 추천)
   - 강의 자료 요약 AI (PDF·노트·슬라이드 업로드 → 핵심 요약 + 퀴즈/포인트 자동 생성)
   - 스터디 플래너 챗봇 (시간표·시험 일정 기반 학습 스케줄·우선순위 추천)
   - 학생 맞춤형 자료 검색 AI (라이브러리·논문·강의자료에서 필요 정보만 추출)

## 핵심 정리

- **문제 선정 논리가 견고**: 3개 후보를 "결과/한계"까지 분석해 비교한 뒤 2번을 선택하고, 기존 해결책(회의·구글드라이브·범용 PM툴)의 한계를 "학교 데이터 미연결 + LLM 자동지원 부재"로 명확히 짚음.
- **차별점은 "학교 특화 데이터 연결"**: 범용 도구와의 결정적 차이를 캠퍼스 데이터 통합으로 잡음.
- **흥미로운 긴장**: 최종 문제는 ②협업이지만, 아이디어 스케치의 4개 기능은 ①자료관리·③AI실습까지 폭넓게 포괄 — 범위가 협업보다 넓어짐(아래 보완 참고).

## 평가 관점 메모 (멘토링/심사용)

**강점**
- 문제 후보 도출 → 선택 근거 → 기존 해결책 한계 → 아이디어 스케치로 이어지는 **기획 흐름이 교과서적으로 깔끔**.
- 실재하는 레퍼런스(경희대 ChatKHU)를 들어 실현 가능성을 뒷받침.
- "학교 데이터와의 연결"이라는 구체적 차별화 포인트 보유.

**보완·질문거리**
- **범위 정합성**: 최종 문제는 "팀 협업"인데 아이디어는 4-in-1 플랫폼으로 확장됨 — 캡스톤 규모상 **협업 도우미 1개로 좁혀 MVP화**하는 편이 현실적. 어떤 1개를 먼저 만들 것인지 결정 필요.
- **데이터·연동 현실성**: 학교 LMS/도서관/강의자료에 대한 접근 권한·API가 실제로 확보 가능한가? (대학 IT 협조가 최대 변수)
- 개인정보(성적·시간표·과제) 취급 시 보안/동의 설계 필요.
- 성공 지표(협업 효율을 무엇으로 측정?)와 기존 PM툴(Notion·Jira) 대비 "AI가 추가로 주는 가치"의 정량화가 다음 과제.
